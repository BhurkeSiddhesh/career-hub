Full Pitch
STAR
Hard Q&A
My Notes
"I built a multi-agent AI decisioning system for a US grocery chain's refund operations. The problem wasn't just cost — it was that 10 reps were making 10 different decisions on the same case. I fixed that by building four AI agents that reason together and produce a consistent, explainable recommendation every time."

The Actual Problem

Jewel-Osco had customer service reps manually approving or denying refund requests. No framework, no audit trail. Reps were inconsistent — same household, different outcomes depending on the rep. The business couldn't scale it and couldn't defend decisions to leadership.

Why I Started With Data, Not the Model

Before building any agent, I spent weeks creating three separate datasets — household level, order level, and item level. The hypothesis: refund fraud risk lives in the combination of signals. A high-value customer ordering a known problem SKU in an unusual pattern is a different risk profile than either signal alone. I created 25+ KPIs across these three layers.

The Percentile Trick — Key Engineering Insight

Raw KPI values mean nothing to a reasoning model. A household spend of $3,400 over 6 months — is that high or low? I converted every KPI to a percentile score against the population. The model doesn't see "$3,400" — it sees "household spend = 94th percentile = top-tier customer in last 6 months." The model can now reason contextually.

The Four-Agent Structure

Three specialist agents each analyzed the refund request from one lens — EntityProfiler (who is this customer?), ContextAnalysis (what does this order look like?), RiskSignal (what is the risk profile of this SKU?). Each produced a structured risk summary. A fourth DecisionAgent synthesised all three into a final call with written rationale. We used GPT-o3 for the decision layer — a reasoning model, not a generation model.

Human-in-the-Loop — Why It Was Essential

Reps couldn't trust a system they couldn't interrogate. The agent recommendation was visible alongside reasoning. Override rate started at ~40% in week one and dropped to under 15% by week six.

Results

500 cases, 5 stores — 11.3% refund cost reduction, 28% productivity savings, 89.16% decision consistency, 94.30% primary-driver reliability.

SituationJewel-Osco reps manually processed refunds with no framework — same case, different decisions across reps, no audit trail, uncontrolled cost.
TaskBuild an automated decisioning system that was explainable enough for reps to trust and consistent enough to defend to leadership.
ActionBuilt three separate feature datasets (household/order/item), percentile-transformed all 25+ KPIs for reasoning model interpretability, designed a 4-agent pipeline where specialist agents feed a decision agent using GPT-o3. Implemented human-in-the-loop for trust calibration.
Result11.3% refund cost reduction, 28% time savings, 89.16% decision consistency, 94.30% primary-driver reliability across 500 PoC cases. Rep override rate dropped from 40% to under 15% over 6 weeks.
Why not just use heuristics instead of building an agentic system?
Heuristics work when decision rules are static and knowable upfront. The refund problem has neither property. The risk signal isn't in any single feature — it's in the interaction between features. A rule like 'if household spend is high, approve' would approve a high-spend customer actively gaming the system. You'd need hundreds of combined rules to capture the interaction space, and the moment fraud patterns shift, every rule breaks. The agentic approach lets each specialist agent reason from its own context, and the decision agent reconciles them. We validated this on a holdout set with 3 novel fraud patterns not seen in training — heuristics would have missed all three, the agent system flagged two of them.
What was your eval framework? How did you actually measure this?
Decision Consistency (89.16%): Duplicated 200 cases with slight surface variations. Ran both versions through the system, checked whether final decision and primary driver matched. 89.16% match rate. The 10.84% inconsistency was mostly on genuinely borderline cases.

Primary-Driver Reliability (94.30%): Had 3 senior reps manually label the most important risk factor for 100 cases. The agent's stated primary driver matched 94.3% of the time.

Cost Delta (11.3%): Simulated full 500-case set with agent decisions vs. historical rep decisions. Mapped both to known fraud outcomes from chargeback data.
What challenges did you face?
Data linkage: Household, order, and item data lived in three separate systems with no shared unique key. Probabilistic customer matching — ~12% of cases had incomplete linkage, got flagged for mandatory human review.

Percentile calibration drift: Percentiles on a static window go stale. Built rolling 90-day windows, refreshed weekly.

Agent disagreement handling: Went through 11 prompt iterations before adjudication logic felt consistent.

Rep resistance: Some reps felt the system evaluated their past decisions. Framed explicitly as a support tool, not a performance tool.
How would you have done this differently?
Structured output schemas: Each agent produced semi-structured text. I'd now force strict JSON — risk_score, top_3_signals, confidence_level, recommended_action. Cleaner inputs make the decision agent more reliable.

Smaller fine-tuned decision agent: GPT-o3 was expensive at scale. With 500+ labeled decisions, I'd fine-tune a smaller model on the decision layer.

Earlier co-design with reps: I built the KPI framework from data analysis without enough rep input. They identified two patterns I'd missed — 'same-item repeat return' and 'short-interval multi-SKU return.'
What was success for you in this project?
The 11.3% cost number is what goes on the slide — but that wasn't what told me the project had worked. The signal was the rep who was most skeptical at the start — 9 years at Jewel-Osco, openly dismissive in the first two sessions. In week five, she used the agent's reasoning to explain her own decision to a supervisor. She said 'the system flagged it as low risk because of household spend and order history, and I agreed with that reasoning.' The cost number is a consequence. The real achievement was making a system explainable enough that an experienced operator chose to own its reasoning as her own.
What are the three specialist agents and what does each one do?
EntityProfiler: Household-level signals — total spend percentile, order count, days as customer, loyalty history. Answers "is this a trustworthy customer?"

ContextAnalysis: This specific order — order value relative to their history, timing, pattern vs. typical behaviour. Answers "does this order look normal for this customer?"

RiskSignal: The item being returned — historical fraud rate for the SKU or product category, return patterns. Answers "is this a high-risk item?"

The fourth DecisionAgent reads all three and makes the final call with written rationale. The separation makes reasoning auditable and easy to debug.

SKU Segmentation Engine

Automated monthly SKU segmentation for Brenntag (NL). Aug–Sep 2025.

Client Brenntag, NL
Type Unsupervised ML · Clustering

88%

3 days

Full Pitch
STAR
Hard Q&A
My Notes
"I solved a problem that makes most segmentation projects fail in production — cluster instability across time. Brenntag needed monthly SKU segments that planners could act on repeatably. Standard clustering gives you different shapes every month. I built a rolling-profile approach that makes clusters stable enough to attach business playbooks to."

The Real Problem (Not the One in the Brief)

The stated ask was 'segment our SKUs.' The actual problem: they'd tried this before and it never stuck — clusters reshuffled every month because clustering was done on point-in-time data. A planner cannot build a playbook on a cluster that reshuffles every month.

Why This Was Hard

Chemical distribution SKUs have high demand variability — volumes spike for industrial procurement cycles, crash during plant shutdowns, recover. Monthly data is noisy. The naive fix (moving average smoothing) loses the trend signal you need to distinguish genuinely declining SKUs from cyclically slow ones.

Rolling 12-Month Behavioral Profiles

Instead of clustering on point-in-time KPIs, I computed each SKU's behavioral fingerprint across trailing 12 months: average monthly volume, coefficient of variation (demand stability), order frequency, margin contribution, and trend slope. These describe the SKU's character, not its state. Applied PCA to reduce multicollinearity, then K-means on the stable profile space.

Why K=5

Two reasons. The business already had five planning playbooks — fast mover, standard, slow mover, critical low-stock, discontinue review. I started from that constraint and validated with silhouette scores (0.61 vs 0.48 for K=4 and 0.57 for K=6).

Operational Delivery

Automated monthly pipeline — compute rolling KPIs on the 1st of each month, run PCA + K-means, map clusters to business actions via heuristic thresholds, export to AWS S3 + Power BI. Planners get their segment assignments every month without touching the model.

SituationBrenntag needed reliable monthly SKU segmentation across 4 EU plants — previous attempts failed because cluster membership reshuffled every month, making business playbooks impossible to attach.
TaskBuild a segmentation engine where clusters are stable enough month-over-month that planners can apply consistent actions to each segment.
ActionComputed 12-month rolling behavioral profiles per SKU rather than point-in-time KPIs, applied PCA + K-means (K=5), mapped clusters to 5 business actions via heuristic thresholds, automated full pipeline to AWS S3 + Power BI.
ResultAutomated monthly segmentation replacing 3 days of manual planner work. Cluster stability (ARI) of 0.81 month-over-month. 88% planner agreement on cluster assignments in validation.
Why not build a classification model with fixed thresholds and reuse it?
A classification model requires labeled training data — someone has to define 'high priority' for each plant's specific KPI distributions. Brenntag has ~2,000 plants globally. You'd need local planner input for 2,000 plants, ongoing label maintenance, and retraining whenever a plant's product mix changes. Clustering discovers natural groupings without requiring pre-labeled ground truth. The business action mapping is done once at the global level and interpreted locally. A valid hybrid: run clustering once to discover natural groupings, then train a lightweight classifier on those cluster labels to score new SKUs in real-time.
How could you have made the clusters more actionable?
Constrained clustering: K-means optimizes for within-cluster variance — it doesn't guarantee groups are maximally useful for business decisions. Add constraints: 'no cluster should have fewer than 50 SKUs', 'each cluster must have a distinct action implication.'

Co-design upfront: The planners identified that 'order frequency' mattered more than I had weighted it — a SKU ordered once a year in large volume is operationally very different from one ordered weekly in small quantities, even if total annual volume is identical.

Interpretability layer: Each cluster got a numeric label. I would add auto-naming — given each cluster's centroid profile, generate a plain-language descriptor: 'high-velocity, stable demand, premium margin.'
What is Adjusted Rand Index and why did you use it?
ARI measures how similar two clusterings are to each other. Score of 1.0 = cluster membership is identical between runs. Score of 0 = overlap no better than random. I used it to measure month-over-month cluster stability — did the same SKUs end up in the same segments in October as September? We achieved 0.81, meaning the vast majority stayed in the same cluster across consecutive months. This matters because Brenntag's previous segmentation failed in production — not because the clusters were mathematically wrong, but because they reshuffled every month. ARI gave us a single number to prove that problem was solved. We targeted above 0.75 and hit 0.81.

Full Pitch
STAR
Hard Q&A
My Notes
"Fashion demand forecasting is one of the hardest forecasting problems because the signal you most need — trend — is exactly what breaks standard models. I built a hybrid system that combined Prophet's trend decomposition with Gradient Boosting's feature learning, and layered in SKU behavioral clusters to give the model 'this shoe is seasonal' as structured context."

Why Fashion Is Hard

Most forecasting literature is built on FMCG — stable products, long history, predictable seasonality. Fashion footwear has short product lifecycles (many SKUs live 1–2 seasons), trend-driven demand spikes that don't recur, and high cannibalization between new and old styles.

The Cluster Input — Key Architectural Choice

I ran a separate clustering exercise on demand shape patterns — not demand level. Found 6 shapes: monsoon peak, festive spike, steady-state formal, athleisure year-round, summer sandal, winter boot. Each SKU was assigned a cluster label based on its historical demand shape. This label went into the forecasting model as a categorical feature — telling the model 'this SKU follows the monsoon pattern' as structured input.

The Prophet Hybrid

Prophet is good at decomposing a time series into trend + seasonality + holiday effects. It's not a great direct forecasting model for erratic data. I ran Prophet on each SKU's historical data, extracted its forward-projected trend and seasonality components as input features to the Gradient Boosting model. GBM then learned which Prophet signals were reliable for which cluster types.

M+2 Horizon and Phased Deployment

We forecast the 3rd month from today (M+2). Top 200 SKUs (80%+ of revenue) hit MAPE under 20% — the operational threshold for automated reorder. We shipped those first. The remaining ~800 SKUs had higher error — rather than deploy bad forecasts, we kept those on planner judgment with the model output as a reference signal.

SituationMetro Brands had 15%+ stockout rate on popular styles and overstock on slow movers — inaccurate demand forecasts across 320+ stores and 1,000 SKUs.
TaskBuild a demand forecast accurate enough to drive automated reorder point calculations for fashion footwear, where erratic trends break standard models.
ActionCreated 6 demand-shape clusters as model features, used Prophet decomposition outputs as hybrid inputs to Gradient Boosting, forecast M+2 horizon, phased deployment starting with top 200 SKUs meeting MAPE <20%.
Result15% improvement in overall forecast accuracy. Top 200 SKUs at MAPE <20% deployed to automated reorder. Client reported ~12% stockout reduction and ~8% dead stock reduction 8 weeks post-deployment.
What did you try for the remaining 800 SKUs and why didn't it work?
More features: Added price elasticity, promotional flags, store-region variables. Marginal improvement, ~1-2 MAPE points. The fundamental problem is data sparsity: many SKUs had 12-18 months of monthly data — 12-18 training observations. No feature engineering fixes that.

Global model: A single GBM trained across all 1,000 SKUs with SKU cluster embeddings. Helped mid-tier SKUs (~3 MAPE points for ~200 SKUs) but barely moved the long tail.

Intermittent demand models (Croston's): Mixed results. Helped for genuinely intermittent SKUs but not for erratic-but-not-zero patterns.

Honest answer: For new fashion SKUs with limited history, the best approach is analogical reasoning — find historically similar SKUs at the same lifecycle stage and use their trajectory. I didn't have time to build that, and it's what I'd tackle first in a second phase.
How did you define success for this project?
Model criterion: MAPE <20% for top 200 SKUs — agreed upfront because below 20%, the expected cost of forecast-driven inventory errors was lower than manual planning hours. We hit this for 82% of the top 200 SKUs.

Business criterion: ~12% reduction in stockout events and ~8% reduction in dead stock value. I want to be careful — there were other supply chain changes happening simultaneously.

Emergent signal: The planning team started using the forecast error distribution to set safety stock buffers. High-uncertainty SKUs got larger buffers. They derived this themselves from model outputs — the best signal that a model has been genuinely integrated into decision-making.

Full Pitch

Hard Q&A
My Notes
"I built a supply network simulation for P&G Japan that found $1M in logistics savings — not by changing routes or carriers, but by identifying that the company had a commercial lever it was not using: flexible delivery date windows in customer contracts that had never been systematically exploited for consolidation."

The Problem Framing

P&G Japan was dispatching partial truckloads because orders were scheduled to rigid Required Delivery Dates. Some customer contracts had RDD flexibility — ±1 or 2 days — that the logistics team never used because there was no tool to identify which orders could be pulled forward without violating the customer commitment.

How the Simulation Worked

Layer 1 — Flexibility Filter: Parse every open future order against its contract RDD tolerance. Only orders with documented RDD flexibility were included. This conservative filter excluded ~60% of order volume but made the output defensible to procurement and commercial teams.

Layer 2 — Consolidation Engine: For each consolidation-eligible order, model whether pulling it to today's shipment would bring the truck above 85% fill rate (threshold for full-truck economics).

Layer 3 — Cost Model: Calculate the unit cost delta between consolidated and unconsolidated dispatch, multiplied by order volume, summed by customer.

Operational Output

The simulation ran daily. Each morning, planners saw a ranked list: 'these 8 orders should be held until tomorrow's dispatch to form a full truck for Customer X, saving ¥XXX per order.' It was a decision tool, not a reporting tool.

SituationP&G Japan was dispatching partial truckloads on fixed RDD schedules — leaving truck capacity underutilized because no system identified when consolidation was contractually possible.
TaskBuild a simulation that identifies order consolidation opportunities using RDD flexibility windows, quantifies savings per customer, and gives planners daily actionable recommendations.
ActionBuilt a three-layer simulation: RDD flexibility filter from contract data, consolidation engine modeling fill-rate impact, cost model calculating savings per consolidation action. Output as daily planner decision tool.
Result$1M savings opportunity identified across top 10 customers. Piloted with highest-opportunity customer — actual savings matched model prediction within 8% variance over 4-week pilot.
How did you actually model the truck fill calculation?
Truck capacity is multi-dimensional — weight, volume, and for some P&G categories, hazmat classification compatibility. We simplified to weight-based fill for the initial model, which was the binding constraint for ~70% of routes. Volume constraints were relevant for bulkier SKUs (detergent, fabric care) but we flagged those for manual review. The 85% fill threshold came from P&G's own carrier contracts — a rate break at 85% fill. Below that, spot rates apply. Above it, contracted rates apply. The $1M calculation was based on the contracted vs. spot rate differential multiplied by the consolidatable order volume.
How did you validate the $1M figure was not just theoretical?
Three validation steps: (1) Ran the model on 3 months of historical order data where actual dispatch outcomes were known — correctly identified consolidation opportunities that planners had manually executed in 76% of cases. (2) The $1M was conservative — we only included orders with documented RDD flexibility (40% of order volume). The real opportunity is probably 1.5-2x larger. (3) Piloted with one customer for 4 weeks — actual savings matched model prediction within 8% variance.
What were the data challenges?
The RDD flexibility data was the hardest part. It lived in customer contract documents — PDFs and Excel attachments in an email archive, not in the order management system. I had to work with the commercial team to extract flexibility windows and build a reference table. This took 3 weeks. Second challenge: order amendment rate of ~18% within the 48-hour window before dispatch. The model re-ran twice daily. In production, this would need a webhook-triggered recalculation whenever an order amendment came in.

Full Pitch

Hard Q&A
My Notes
"I built a shipment consolidation recommendation system for Schneider Electric's North American logistics. The core insight was that the biggest savings opportunity was not in route optimization or carrier negotiation — it was in the data itself. Reason codes were labeling non-urgent shipments as urgent, hiding a large consolidation pool that nobody could see."

The Actual Problem

Schneider was running too many partial or singleton shipments in NAM. The logistics team knew costs were high but couldn't identify which shipments were genuinely time-critical vs. which ones had been labeled urgent out of habit or to avoid the consolidation queue.

Reason Code Analysis — Where the Real Work Was

Mapped all shipment reason codes into three categories: hard constraints (truly non-consolidatable), soft constraints (planner preference, habit-based), and consolidatable. Then linked reason codes back to actual delivery outcomes. An 'urgent delivery' code that resulted in a shipment arriving 3 days after the standard lead time is a de facto soft constraint. About 30% of 'urgent' tagged shipments had no evidence of urgency in delivery records.

Multi-Interval Design

Segmented customers by shipment frequency and volume, then modeled the minimum holding period to reach 85% truck fill for each segment. High-volume customers got daily consolidation windows. Mid-volume got weekly. Low-volume got bi-weekly.

SituationSchneider's NAM logistics had high per-shipment costs from frequent partial loads — reason code inflation was masking a large consolidation opportunity that planners couldn't see.
TaskIdentify which shipments were genuinely non-consolidatable vs. soft-constrained, and build a recommendation system for optimal consolidation windows per customer.
ActionCategorized reason codes into hard/soft/consolidatable via delivery outcome linkage. Segmented customers by volume/frequency. Built multi-interval consolidation recommendation engine with per-shipment savings calculation.
ResultIdentified 30% of 'urgent' shipments as soft-constraint, unlocking a significant consolidation pool. System gave planners real-time cost signal on dispatch decisions across NAM market.
How did you determine which reason codes were genuinely urgent vs. soft constraints?
Two-step process. First, definitional mapping — worked with operations managers to categorize reason codes by business intent. Second, outcome validation — for ambiguous codes, linked each shipment to its actual delivery record and checked whether the shipment actually needed to arrive on the day it was dispatched. Test: did the customer use the product within 24 hours, and was there a documented exception reason in the CRM? Shipments where delivery arrived on the standard window date with no exceptional reason were reclassified as soft constraints. 30% reclassification rate was conservative enough that the business accepted it as a reasonable lower bound.
What was the hardest data problem?
Cross-system record linkage. The shipment management system, CRM, and delivery tracking system did not share a common customer identifier. Linking a shipment's reason code to its delivery outcome to its CRM exception record required fuzzy matching on customer name and delivery address — no clean join key existed. About 15% of records couldn't be reliably linked and were excluded. I flagged this — if Schneider standardized their customer identifiers across systems, analysis coverage would improve significantly.

Full Pitch

Hard Q&A
My Notes
"I was embedded with Google's EMEA analytics team to build the measurement infrastructure that drove 21% quarter-over-quarter active user growth across markets. The core problem was not data — it was that the existing global dashboard averaged out market-specific dynamics, making it impossible to know what was actually working where."

What the Problem Actually Was

Google had a unified EMEA dashboard showing aggregate SEO metrics. Germany looked fine. Middle East looked fine. But averaging them masked the fact that Germany's user growth had stalled for two quarters while Middle East was growing faster than anyone realized. And no one knew which SEO interventions were causing what — because the attribution model was last-touch, which significantly undervalued early-journey content discovery pages.

Market-Specific KPI Frameworks

Built individual KPI sets for each major EMEA market. The methodology was correlation analysis — for each market, which leading SEO indicators had the highest 4-week leading correlation with active user growth? These varied substantially. In mobile-first markets, crawl efficiency and page speed were the strongest predictors. In desktop-heavy markets, content depth metrics and SERP feature coverage mattered more.

Attribution Model Rebuild — Markov Chain

Replaced last-touch attribution with a data-driven multi-touch model using Markov chain attribution. Mapped user acquisition journeys and calculated each touchpoint's removal effect (what % of conversions would be lost if this touchpoint did not exist?). Revealed that discovery-phase content pages, which got zero credit under last-touch, were critical in 40% of acquisition journeys.

SituationGoogle EMEA had a global SEO dashboard that averaged out market-specific dynamics — misallocating resources and undervaluing organic channel contribution due to flawed last-touch attribution.
TaskBuild market-specific KPI frameworks, a multi-touch attribution model, and OKR evaluation infrastructure that accurately measured SEO's contribution to active user growth per market.
ActionRan correlation analysis to identify market-specific leading indicators. Rebuilt attribution using Markov chain multi-touch model. Built quarterly OKR impact tracking with retrospective accuracy measurement.
Result21% Q/Q active user growth across EMEA. Attribution shift revealed discovery content driving 40% of journeys unrecognized. OKR prediction accuracy improved from ±30% to ±12% over 6 quarters.
How does Markov chain attribution actually work?
Model each user's journey from first SEO touch to becoming active as a sequence of states (touchpoints). Use historical journey data to estimate transition probabilities between states. Then, for each touchpoint, calculate the removal effect: if you delete all journeys containing this touchpoint and re-run the model, what % of conversions disappear? A touchpoint with a 35% removal effect is responsible for 35% of conversions. Sum removal effects, normalize to 100% = attribution weights. Why this over simpler models? Linear attribution says every touch is equal. Time-decay says recent touches matter more. These are arbitrary assumptions. Markov chain derives attribution from actual observed journey patterns — it's empirical, not assumed.
21% Q/Q sounds very high — is that really attributable to your work?
Honest answer: no, not entirely, and I would never claim it was. The 21% Q/Q growth had multiple drivers — product improvements, marketing spend, competitive dynamics, seasonality. What I can specifically attribute: the Markov chain attribution model revealed that discovery content was driving 40% of acquisition journeys but getting near-zero credit under last-touch. This caused a resource reallocation toward that content category in Q3. In markets where that reallocation happened, we measured incremental impression and click growth from the newly resourced pages — that specific increment was trackable to the attribution-driven investment decision. I'm comfortable saying my work was a meaningful contributor, and the methodology changes improved the team's ability to make better decisions.

Gaming Platform Revenue Analytics

Causal inference / PSM for Sony PlayStation. Oct 2021 – Jan 2022.

Client Sony PlayStation, IN
Type Causal Inference · PSM
8%
Causal revenue uplift
23%
Naive (biased) estimate
15pp

2.8x

Full Pitch
STAR
Hard Q&A
My Notes
"I used propensity score matching to measure the causal revenue impact of player engagement for PlayStation — and the most important finding was that the naive number was wrong. The obvious comparison showed 23% revenue difference between high and low engagement players. The causal estimate was 8%. The 15-point gap is selection bias, and betting a product strategy on the wrong number is an expensive mistake."

Why the Naive Analysis Misleads

High-engagement players also spend more — but that's correlation, not cause. Players who are intrinsically motivated to spend more on games also tend to play more. If you don't account for that, you'll overestimate the revenue impact of an engagement intervention.

Propensity Score Matching

I used pymatch to compute each player's propensity score — the probability of being a high-engagement player given observable characteristics: game genre preference, device type, geographic region, account age, historical average spend, and session frequency. Players with similar propensity scores got matched — one high-engagement (treatment), one medium-engagement (control). The revenue comparison within matched pairs isolates the engagement effect.

The 2D Behavioral Matrix — The Strategic Finding

Segmented players on two dimensions: playtime quartile and spending quartile. The highest revenue uplift potential was in the high-playtime/low-spend quadrant — engaged players who were not monetized. This completely shifted the product team's investment thesis: from 'acquire more high-spenders' to 'convert engaged non-spenders.'

SituationPlayStation's revenue analytics was based on naive comparisons — overstating the revenue impact of engagement interventions by 15+ percentage points due to selection bias.
TaskBuild a causal framework to measure the true revenue impact of engagement on player spending, controlling for pre-existing player characteristics that confound the naive comparison.
ActionApplied propensity score matching (pymatch) to create comparable treatment/control groups on observable player characteristics. Built 2D behavioral matrix (playtime x spending quartiles). Measured revenue uplift within matched pairs.
Result8% causal revenue uplift identified vs. 23% naive estimate. High-playtime/low-spend quadrant identified as highest-ROI targeting opportunity. Product strategy shifted from high-spender acquisition to engaged-non-spender conversion.
Why PSM and not just running an A/B test?
A/B test would have been better — it's the gold standard for causal inference because randomization handles both observed and unobserved confounders. But this was retrospective analysis on existing player data. There was no experiment to run. PSM is the right tool for observational causal inference: match on observables to approximate the randomization you'd get from an experiment. Limitation: PSM only controls for observed confounders. If there are unobserved player characteristics — disposable income, gaming motivation type — that simultaneously affect engagement and spending, PSM doesn't remove that bias. We ran a Rosenbaum bounds sensitivity analysis — the finding would only be invalidated by an unobserved confounder creating a 2.8x difference in treatment odds. Given we controlled for genre, device, region, account age, session frequency, and historical spend, that's implausibly large.
How did you validate the quality of your matching?
Standard covariate balance checks. Before matching, SMDs on key covariates were large: account age 0.42, historical spend 0.61, session frequency 0.55. Falsification test: ran analysis on a variable that should have zero causal relationship with engagement — console purchase date. Model showed 0.3% — effectively zero.
What was the 2D behavioral matrix and what did it show?
Segmented all players simultaneously on two axes: playtime quartile (Q1=lowest 25% play hours, Q4=highest) and spending quartile. Creates a 4x4 grid of 16 cells. The strategically important cell was high playtime + low spend — players in Q3-Q4 playtime but Q1-Q2 spending. These are deeply engaged players who are not monetized. Why Costs Were Opaque

MARS had quarterly cost reports showing costs went up by X%. But nobody could say why. Was it raw material prices? Volume mix shifts? Plant efficiency degradation? Finance teams produced narrative explanations, but they were largely post-hoc rationalisation.

Why a Neural Network Over Tree Models

Explainability via Integrated Gradients requires a differentiable model. Tree models are not differentiable — SHAP is the standard attribution method there, which is post-hoc and approximate. I chose PyTorch deliberately because I needed exact gradient-based attribution.

Integrated Gradients — How to Explain It

Imagine you want to understand why a cost went from ₹100 to ₹115. Integrated Gradients computes, for each input feature, how much of that ₹15 increase it's responsible for — by tracing the model's computation path from a baseline to the actual input. It satisfies the completeness axiom: attributions sum exactly to the total output change. So if raw materials drove ₹8, volume drove ₹4, and logistics drove ₹3 — those add up to exactly ₹15. No unexplained residual.

The Scenario Simulation Layer

Once the model was trained, I built a Streamlit simulation interface where business users could dial inputs — raw material price +10%, plant capacity -5%, volume mix shift — and see the predicted cost impact with full driver attribution. This moved the tool from diagnostic to prescriptive.

Why Integrated Gradients over SHAP?
SHAP is more widely used, and for tree models (TreeSHAP), it's the right tool — fast, exact, well-validated. For neural networks, KernelSHAP applies, which is an approximation computed by sampling feature subsets. Integrated Gradients is exact for differentiable models. It satisfies the completeness axiom — attributions sum exactly to the model output — which KernelSHAP doesn't always guarantee. For a business context where someone is pointing at a number and saying 'raw materials caused ₹8 of our ₹15 cost increase,' mathematical exactness matters. You don't want the CFO asking why the attributions don't add up to the total.

The most predictive features were not the obvious ones. Why CatBoost + XGBoost Ensemble

CatBoost handles high-cardinality categoricals natively without one-hot encoding. XGBoost was stronger on numerical features. The ensemble (weighted average, 60/40 XGBoost/CatBoost) outperformed either model individually by ~3 F1 points.

Weekly scoring pipeline — every Monday, all open customer orders for the next 12 weeks were scored. 78% accuracy doesn't sound impressive — how did you justify this for production?
Accuracy is the wrong metric for imbalanced classification problems. Historical data had ~78% on-time fulfillment — a model that predicted 'always fulfilled' would hit 78% accuracy without any intelligence. That's the baseline. The metrics that mattered: precision and recall on the failure class, and cost-weighted F1. At our operating threshold, the model caught 71% of all fulfillment failures (recall) with 58% precision. Without the model, planners were catching perhaps 20-30% of failures through intuition. 71% recall was a genuine step-change in risk visibility.
Walk me through what actually happened with the PSI spike.
PSI measures how much a feature's distribution in production has shifted compared to training. Thresholds: below 0.1 = stable, 0.1–0.2 = slight drift worth watching, above 0.2 = significant drift and retraining trigger. For roughly seven months, raw material availability PSI was sitting at 0.08. Then a supplier disruption hit. In a single monthly PSI check, that feature jumped from 0.08 to 0.34 — nearly four times the retraining threshold. We retrained on a rolling 18-month window that included the disruption period. Performance was restored within three percentage points. The planners never noticed a quality issue because we caught and fixed it proactively. That's the entire point of monitoring — you catch it before the business is affected.
How did you handle class imbalance?
SMOTE-ENN: Generated synthetic failure cases by interpolating between existing failure cases in feature space, combined with edited nearest neighbors to remove noisy borderline cases.
Class weights: CatBoost native class_weights; XGBoost scale_pos_weight — both set to reflect inverse class frequency (~4:1).
Threshold calibration: Used Platt scaling to calibrate probability outputs, then chose the threshold on validation set that maximized cost-weighted F1. Operating threshold ended up around 0.35 rather than 0.5, reflecting the asymmetric cost structure.

The most important 'dataset' for this project. Simulates exactly what happened in your MARS project when the supply chain disruption hit. Dashboard goes from all-green to red at batch 15. This is the interview demo.
1
Build the drift metrics engine (drift_watch/metrics.py)PSI (bins-based, handles log(0)). KS test (p-value < 0.05 = drifted). Jensen-Shannon divergence. compute_all_metrics() returns a DataFrame covering all numeric features. Thresholds: PSI < 0.1 stable, 0.1–0.2 slight drift, > 0.2 = retrain.
2
Build the FastAPI logging endpoint (api/main.py)POST /log receives batch features, computes metrics vs. reference distribution, persists to SQLite, triggers Slack alert if any feature PSI > 0.2.
3
Build the Plotly Dash monitoring dashboardPSI heatmap (feature × batch, green→red colour scale). Timeline chart for selected feature. Auto-refreshes every 30 seconds. Model selector dropdown.
4
Write the simulation script (demo/run_simulation.py)The interview demo: loads 30 batches, POSTs them to the FastAPI endpoint one by one with 0.5s delay, then opens the dashboard. Batches 0–14 = green. Batch 15 = red PSI spike. Slack alert fires. This is what you demo live.

The One Rule That Matters Most

Write the README before you write the code. Not a placeholder — a real README describing what the finished project does, with the architecture diagram and the results table (with TBD values you fill in later). This forces clarity. The projects that fail are the ones where the builder was unclear on the output from the start.
